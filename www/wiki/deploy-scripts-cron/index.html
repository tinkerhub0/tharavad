Deploy scripts & cron

This guide teaches you how to clone a repo from GitHub, write a shell script that pulls and restarts your app, and schedule that script to run automatically using cron — so your server always stays up to date without you touching it.

Two-page series

This is part 1 of 2. It covers git setup, deploy scripts, and cron scheduling. Part 2 covers how Caddy works as a reverse proxy and wires everything into a complete production-ready auto-deploy pipeline.

The big picture

Before touching a single command, understand what you're trying to build. The goal is a server that automatically keeps itself up to date with whatever you push to GitHub. The full chain looks like this:

1
You push code to GitHub
Your local machine pushes a commit. The new code now lives on GitHub's servers.
2
cron wakes up on a schedule
A background scheduler on tharavad runs your deploy script at a fixed interval — say, every 5 minutes.
3
The deploy script runs
The script pulls the latest code from GitHub, installs any new dependencies, and restarts the server process.
4
Your site is updated
Visitors hitting your subdomain see the new version within minutes of your push. No manual SSH required.

You'll set this up piece by piece. First git, then the deploy script, then cron.

Part 1 — Setting up git and cloning your repo

What is git clone?

When you run git clone, git downloads the entire history of a repository from a remote server (GitHub, in this case) onto your local machine — or in this case, your tharavad shell. The result is a folder containing all your files plus a hidden .git/ directory that git uses to track changes.

SSH access to GitHub from tharavad

You have two options for authenticating to GitHub from tharavad: HTTPS (with a personal access token) or SSH keys. SSH is strongly recommended — it doesn't expire, you never type a password, and it's what git was designed for.

Generate a key specifically for tharavad → GitHub

You're already on tharavad via SSH. Now you need a separate keypair that lets tharavad authenticate to GitHub. This key lives inside your tharavad shell, not your local machine.

# Run this while SSHed into tharavad
ssh-keygen -t ed25519 -C "tharavad-deploy" -f ~/.ssh/github
ssh-keygen
The key generation utility. Generates a matching pair of private + public key files.
-t ed25519
The key type. ed25519 is a modern elliptic curve algorithm — short, fast, and secure.
-C "tharavad-deploy"
A human-readable comment (-C) to label this key. Helps you identify it later in GitHub's key list.
-f ~/.ssh/github
The output file path. This creates ~/.ssh/github (private key) and ~/.ssh/github.pub (public key) — separate from your main tharavad key so they don't conflict.

When prompted for a passphrase, press Enter twice to skip it. A passphrase would require you to type it every time — which would break unattended scripts running via cron.

Add the public key to GitHub

# Print your new public key
cat ~/.ssh/github.pub

Copy the entire output. Then in GitHub: Settings → SSH and GPG keys → New SSH key. Give it a title like "tharavad server" and paste the key. Click Add SSH key.

Tell SSH which key to use for GitHub

Create or edit ~/.ssh/config to configure SSH to automatically use your github key whenever it connects to github.com:

~/.ssh/config
Host github.com
    HostName  github.com
    User      git
    IdentityFile ~/.ssh/github
    IdentitiesOnly yes
Host github.com
This block applies any time SSH connects to the hostname github.com.
User git
GitHub's SSH server requires the username git for all connections regardless of your GitHub username. This is correct — don't change it.
IdentityFile ~/.ssh/github
Tells SSH to authenticate using your ~/.ssh/github private key for this host.
IdentitiesOnly yes
Forces SSH to only try the specified key. Without this, SSH might try other keys first and confuse GitHub's server.

Test the connection

ssh -T git@github.com

You should see: Hi yourusername! You've successfully authenticated, but GitHub does not provide shell access.

That message is normal and correct. It confirms the key works. GitHub doesn't give you a shell — it only accepts git operations.

Cloning your repository

Navigate to your home directory and clone your repo using the SSH URL (not the HTTPS one):

# Go home first
cd ~

# Clone with SSH (find this URL on GitHub → Code → SSH tab)
git clone git@github.com:yourusername/yourrepo.git

# The repo is now at ~/yourrepo/
cd yourrepo
ls
git clone
Downloads the full repository — all files, all branches, all commit history — into a new folder in your current directory.
git@github.com:
The SSH address of GitHub. git is the username (always), github.com is the host.
yourusername/yourrepo.git
Your GitHub username and repository name. The .git suffix is conventional — git understands it either way.

Understanding git pull

git pull is the command you'll use in your deploy script. It fetches the latest commits from GitHub and merges them into your current branch. Think of it as "download whatever changed since I last synced."

# Inside your repo directory
cd ~/yourrepo
git pull

# Output when there are new commits:
remote: Enumerating objects: 5, done.
remote: Counting objects: 100% (5/5), done.
Updating a1b2c3d..e4f5g6h
Fast-forward
 index.html | 4 ++--
 1 file changed, 2 insertions(+), 2 deletions(-)

# Output when already up to date:
Already up to date.
Important: always pull from the right branch

By default, git pull pulls the branch you're currently on. Make sure your deploy script explicitly checks out the right branch (usually main or master) so you're always pulling the correct code.

Part 2 — Writing a deploy script

What is a shell script?

A shell script is a plain text file containing a sequence of shell commands. When you run the file, the shell reads it top to bottom and executes each command in order — exactly as if you had typed them yourself. Scripts let you bundle many steps into a single command.

The shebang line

Every shell script begins with a shebang — a special comment on line 1 that tells the OS which interpreter to use:

#!/bin/bash
#!
Called "shebang" (or "hashbang"). When the OS sees this as the first two bytes of a file, it knows the rest of the line is the path to the interpreter.
/bin/bash
The absolute path to the bash executable. This ensures the script runs with bash regardless of which shell the user is currently using.

set -e and set -o pipefail

Add these two lines near the top of every deploy script. They make the script stop immediately if anything goes wrong — which is critical in a deploy context. Without them, a failed git pull might be silently ignored and your script would continue trying to restart a server with stale code.

set -e          # exit immediately if any command fails
set -o pipefail # catch failures inside pipelines (cmd1 | cmd2)
set -e
Enables "exit on error" mode. If any command returns a non-zero exit code (which means failure), the script stops immediately. Without this, bash just logs the error and keeps going.
set -o pipefail
Extends the exit-on-error rule to pipelines. Normally, the exit code of cmd1 | cmd2 is only the exit code of cmd2. With pipefail, if cmd1 fails, the whole pipeline fails.

Variables in bash

Use variables to avoid repeating paths throughout your script. Declare them in UPPER_CASE by convention:

# Define once at the top
APP_DIR="$HOME/yourrepo"
PORT=45320

# Use with $ prefix
cd "$APP_DIR"
VARNAME=value
Assignment. No spaces around the = sign — bash will error if you add spaces.
$VARNAME or "${VARNAME}"
Variable expansion. The $ tells bash to substitute the variable's value. Use double quotes around it to handle paths with spaces safely.
$HOME
A built-in environment variable set by the OS to your home directory path (e.g. /home/jake). More reliable than hardcoding the path.

Killing and restarting a process

When you deploy new code, you need to stop the old server and start a new one. This is where most beginners get tripped up — how do you kill a process you started earlier?

The pattern is: save the process ID (PID) when you start the server, then use that PID to kill it later.

# Start a process and save its PID to a file
python3 -m http.server $PORT &
echo $! > /tmp/myapp.pid

# $! is a special variable — it holds the PID of the last
# background process that was started
& (ampersand)
Runs the command in the background. Without it, the command would block the script forever (the server runs until killed). The & tells bash to start it and immediately move on to the next line.
$!
A special bash variable that contains the Process ID (PID) of the most recently started background process. PID is a number the OS assigns to every running process.
echo $! > /tmp/myapp.pid
Writes the PID to a file. > redirects the output of echo into the file, overwriting it. /tmp/ is a good place for temporary runtime files.

To kill the process on the next deploy:

# Check if the PID file exists
if [ -f /tmp/myapp.pid ]; then
    OLD_PID=$(cat /tmp/myapp.pid)
    # Check if a process with that PID is actually running
    if kill -0 "$OLD_PID" 2>/dev/null; then
        kill "$OLD_PID"
        sleep 1   # give it a moment to clean up
    fi
fi
if [ -f file ]
-f tests whether the path is a regular file that exists. The whole [ ... ] block is a test expression. If true, the then block runs.
$(cat /tmp/myapp.pid)
Command substitution. The $(...) runs the command inside and substitutes its output as a value. Here it reads the PID number out of the file.
kill -0 "$PID"
kill -0 sends "signal 0" — which doesn't actually kill anything. It just checks whether the process is still running. Returns exit code 0 (success) if it is, non-zero if it isn't. Used purely for checking.
2>/dev/null
Redirects "stderr" (file descriptor 2, used for error messages) to /dev/null — a special file that discards anything written to it. This silences the "No such process" error if the PID doesn't exist.
kill "$PID"
Sends SIGTERM (signal 15) to the process — a polite "please stop" signal that gives the process a chance to clean up before exiting.

The complete deploy script

Putting it all together. This script covers the full cycle: pull new code, install dependencies, kill the old server, start the new one.

~/deploy.sh#!/bin/bash
set -e
set -o pipefail

# ── Config ─────────────────────────────────────────────────
APP_DIR="$HOME/yourrepo"
PORT=45320
PID_FILE="/tmp/myapp.pid"
LOG_FILE="$HOME/deploy.log"
APP_LOG="$HOME/app.log"

# ── Logging ────────────────────────────────────────────────
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"
}

log "Deploy started"

# ── Pull latest code ───────────────────────────────────────
log "Pulling latest code from GitHub..."
cd "$APP_DIR"
git fetch origin
git reset --hard origin/main
log "Now at commit: $(git rev-parse --short HEAD)"

# ── Install dependencies (if needed) ──────────────────────
# Uncomment whichever applies to your project:
# npm install --production
# pip install -r requirements.txt --user

# ── Stop old server ────────────────────────────────────────
log "Stopping old server..."
if [ -f "$PID_FILE" ]; then
    OLD_PID=$(cat "$PID_FILE")
    if kill -0 "$OLD_PID" 2>/dev/null; then
        kill "$OLD_PID"
        sleep 1
        log "Killed PID $OLD_PID"
    else
        log "PID $OLD_PID was not running (already stopped)"
    fi
    rm -f "$PID_FILE"
fi

# ── Start new server ───────────────────────────────────────
log "Starting server on port $PORT..."
cd "$APP_DIR"

# For a static site:
python3 -m http.server $PORT >>"$APP_LOG" 2>&1 &

# For a Node app (uncomment and replace above):
# node server.js >>"$APP_LOG" 2>&1 &

echo $! > "$PID_FILE"
log "Server started with PID $(cat $PID_FILE)"
log "Deploy complete. Running at port $PORT"
git fetch origin
Downloads all new commits from the remote (GitHub) without applying them yet. origin is the default name git gives to the remote you cloned from.
git reset --hard origin/main
Forces your local branch to exactly match origin/main. This is safer than git pull in automated scripts — it discards any accidental local changes and guarantees you match GitHub exactly. --hard means "move the branch pointer AND update all files to match."
git rev-parse --short HEAD
Prints the short (7-character) hash of the current commit. Useful in logs to confirm exactly which version is deployed.
tee -a "$LOG_FILE"
tee reads from stdin and writes to both stdout AND a file simultaneously. The -a flag appends to the file instead of overwriting. So echo "message" | tee -a file prints to the terminal and logs to the file at the same time.
>>"$APP_LOG" 2>&1
>> appends stdout to the log file. 2>&1 redirects stderr (file descriptor 2) to wherever stdout (file descriptor 1) is going — meaning both regular output and errors end up in the same log file.

Making the script executable

A file has to be marked executable before you can run it as a command. Use chmod:

chmod +x ~/deploy.sh
chmod
Change file mode (permissions). Every file on Linux has read/write/execute permissions for the owner, group, and others.
+x
Adds the execute bit for all users. After this, you can run the file directly with ./deploy.sh or just ~/deploy.sh.

Test it manually first before setting up cron:

bash ~/deploy.sh

Watch the output. If anything fails, fix it before automating. Check the log file for details:

cat ~/deploy.log

Part 3 — Scheduling with cron

What is cron?

cron is a time-based job scheduler built into every Linux system. It runs in the background as a daemon (a background system process) and wakes up every minute to check if any scheduled jobs need to run. If they do, it runs them. You define the schedule in a file called the crontab (cron table).

cron is the backbone of automation on Linux servers. You use it to run backups, send reports, rotate log files, and — as we're doing here — automatically deploy updated code.

The crontab file

Each user has their own crontab. Edit yours with:

crontab -e

This opens the crontab in your default editor (usually nano or vi). Each line in the crontab defines one scheduled job in this format:

# ┌─────── minute       (0–59)
# │ ┌───── hour         (0–23)
# │ │ ┌─── day of month (1–31)
# │ │ │ ┌─ month        (1–12)
# │ │ │ │ ┌ day of week  (0–7, 0 and 7 are Sunday)
# │ │ │ │ │
  * * * * *  command to run

The fields in detail

FieldRangeSpecial values
minute0–59* = every minute, */5 = every 5 min, 0,30 = at 0 and 30
hour0–23* = every hour, 2 = at 2am, */6 = every 6 hours
day of month1–31* = every day, 1 = 1st of month, 15 = 15th
month1–12* = every month, 1 = January
day of week0–7* = every day, 1 = Monday, 0 or 7 = Sunday

Common schedule examples

ExpressionMeaning
* * * * *Every minute
*/5 * * * *Every 5 minutes
*/10 * * * *Every 10 minutes
0 * * * *Every hour, on the hour
0 2 * * *Every day at 2:00am
0 0 * * 1Every Monday at midnight
30 6 * * 1-5Mon–Fri at 6:30am
0 */6 * * *Every 6 hours

The PATH problem — the most common cron pitfall

This is where almost everyone gets stuck the first time. Your deploy script works perfectly when you run it manually, but fails silently when cron runs it. The reason: cron runs in a minimal environment with a very limited PATH.

When you type commands in your shell, bash knows where to find programs like git, node, python3 because your $PATH lists many directories. Cron's PATH is typically just /usr/bin:/bin — it won't find things installed in /usr/local/bin, ~/.local/bin, or anywhere else.

The fix: set PATH explicitly at the top of your crontab, or use full absolute paths in your script.

# Set PATH at the top of your crontab (add this as first line)
PATH=/usr/local/bin:/usr/bin:/bin:/home/jake/.local/bin
HOME=/home/jake

# Then your job:
*/5 * * * *  /home/jake/deploy.sh >> /home/jake/cron.log 2>&1
Always use absolute paths in cron

Never write ~/deploy.sh in a crontab — the ~ shorthand may not expand correctly. Always use the full path: /home/jake/deploy.sh. Run echo $HOME in your shell to find your exact home directory path.

Capturing cron output in a log

By default, cron discards all output (unless mail is configured, which it isn't on tharavad). You'll have no idea if jobs succeeded or failed unless you redirect output to a file yourself.

# Append both stdout and stderr to a log file
*/5 * * * *  /home/jake/deploy.sh >> /home/jake/cron.log 2>&1

# Check the log anytime
tail -f ~/cron.log
tail -f ~/cron.log
tail prints the last lines of a file. -f means "follow" — it keeps the file open and prints new lines as they're added in real time. Use this to watch your cron job run live. Press Ctrl+C to stop.

Setting up the cron job

# Open your crontab for editing
crontab -e

Add these lines (replace jake with your username, found by running whoami):

crontabPATH=/usr/local/bin:/usr/bin:/bin:/home/jake/.local/bin
HOME=/home/jake

# Pull and redeploy every 5 minutes
*/5 * * * *  /home/jake/deploy.sh >> /home/jake/cron.log 2>&1

Save and exit. Verify it was saved:

crontab -l     # list current crontab

Verifying cron is running your job

Wait a few minutes, then check the log:

cat ~/cron.log

You should see timestamped lines from your deploy script's log() function. If the file is empty or doesn't exist, the job hasn't run yet — wait a full 5 minutes and check again.

What good output looks like

[2025-04-10 14:35:01] Deploy started
[2025-04-10 14:35:01] Pulling latest code from GitHub...
[2025-04-10 14:35:02] Now at commit: a1b2c3d
[2025-04-10 14:35:02] Stopping old server...
[2025-04-10 14:35:03] Killed PID 12345
[2025-04-10 14:35:03] Starting server on port 45320...
[2025-04-10 14:35:03] Server started with PID 12398
[2025-04-10 14:35:03] Deploy complete. Running at port 45320

Only redeploy when there are new commits

Running the full stop/start cycle every 5 minutes even when nothing changed causes unnecessary downtime. You can make the script smarter — only restart if git pull actually fetched new commits:

# Get the current commit hash before pulling
BEFORE=$(git rev-parse HEAD)

# Fetch and reset
git fetch origin
git reset --hard origin/main

# Get the commit hash after pulling
AFTER=$(git rev-parse HEAD)

# Only restart if something changed
if [ "$BEFORE" = "$AFTER" ]; then
    log "No changes. Skipping restart."
    exit 0
fi

log "New commit detected: $BEFORE → $AFTER. Restarting..."
# ... rest of deploy continues

Putting it all together

Here is the final production-ready deploy script with the "only restart on change" optimisation included:

~/deploy.sh#!/bin/bash
set -e
set -o pipefail

APP_DIR="$HOME/yourrepo"
PORT=45320
PID_FILE="/tmp/myapp.pid"
LOG_FILE="$HOME/deploy.log"
APP_LOG="$HOME/app.log"

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"; }

log "── Checking for updates ──────────────────────"
cd "$APP_DIR"

BEFORE=$(git rev-parse HEAD)
git fetch origin
git reset --hard origin/main
AFTER=$(git rev-parse HEAD)

if [ "$BEFORE" = "$AFTER" ]; then
    log "Already up to date ($(git rev-parse --short HEAD)). Nothing to do."
    exit 0
fi

log "New code detected. Deploying $(git rev-parse --short HEAD)..."

# Install deps — uncomment what applies:
# npm install --production --silent
# pip install -r requirements.txt --user -q

if [ -f "$PID_FILE" ]; then
    OLD_PID=$(cat "$PID_FILE")
    if kill -0 "$OLD_PID" 2>/dev/null; then
        kill "$OLD_PID" && sleep 1
        log "Stopped old server (PID $OLD_PID)"
    fi
    rm -f "$PID_FILE"
fi

python3 -m http.server $PORT >>"$APP_LOG" 2>&1 &
echo $! > "$PID_FILE"
log "Server started on port $PORT (PID $(cat $PID_FILE))"
log "── Deploy complete ───────────────────────────"
