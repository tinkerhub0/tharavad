Caddy & the full auto-deploy pipeline

This guide explains how Caddy works as a reverse proxy, why your app can't just bind to port 443 directly, and how to wire everything — git, your deploy script, cron, and Caddy — into a complete pipeline that auto-deploys your code to a live HTTPS URL.

Part 1 — How Caddy works

What is a reverse proxy?

When a visitor types https://myapp.tharavad.xyz in their browser, their browser connects to the tharavad server on port 443 (the standard HTTPS port). Your app is running on port 45320. Someone needs to sit in between and forward the request.

That's exactly what a reverse proxy does. It listens on port 443, accepts the incoming connection, forwards it to your app on port 45320, waits for the response, and sends it back to the browser. From the browser's perspective, it's talking to myapp.tharavad.xyz. From your app's perspective, it's just receiving a regular HTTP request on its local port.

↓
Browser makes a request
GET https://myapp.tharavad.xyz/index.html — TCP connection to the server on port 443
↓
Caddy receives it on port 443
Caddy handles the TLS handshake (decrypts the HTTPS traffic), reads the Host header to determine which app to route to
↓
Caddy looks up routing rules
Matches "myapp.tharavad.xyz" against the routing table created by tinker caddy — finds it routes to localhost:45320
↓
Caddy forwards to your app
Makes a plain HTTP request to http://localhost:45320/index.html on the local network
↑
Your app responds
Returns the file or response on port 45320
↑
Caddy re-encrypts and sends back
Wraps the response in TLS and sends it back to the browser. The visitor sees HTTPS.

Why you can't bind directly to port 443

On Linux, only processes running as root can bind to ports below 1024 (these are called "privileged ports"). Ports 80 (HTTP) and 443 (HTTPS) fall in that range. Since you're a regular user on tharavad without root access, your app simply cannot listen on those ports.

Caddy runs as a system process with root privileges. It owns ports 80 and 443. Your app runs on a high port (like 45320) in user space. Caddy bridges the gap.

What TLS / HTTPS means and why Caddy handles it

TLS (Transport Layer Security) is the protocol that encrypts traffic between the browser and the server — it's what makes HTTPS secure. To serve HTTPS, a server needs a TLS certificate issued by a Certificate Authority (CA) that browsers trust. Getting and renewing these certificates used to be complicated and cost money.

Caddy does all of this automatically. It contacts Let's Encrypt (a free CA) the first time it sees a new subdomain, obtains a certificate, and renews it before it expires. When you run tinker caddy myapp 45320, within seconds Caddy has a valid certificate for myapp.tharavad.xyz and starts serving HTTPS. You don't touch any of this.

What tinker caddy actually does

When you run tinker caddy myapp 45320, the tinker tool talks to Caddy's admin API (which listens locally on the server) and adds a new routing rule that says: "requests for myapp.tharavad.xyz should be proxied to localhost:45320." The rule is applied live — Caddy reloads the config in memory without restarting, meaning no downtime.

One Caddy, many apps

Caddy uses the Host header in every HTTP request to figure out which app to route to. Every HTTP request carries the domain name the browser was trying to reach. This means one Caddy instance can host hundreds of different apps on the same IP address — it just routes each request to the right port based on the subdomain.

Part 2 — The deploy script handles Caddy too

Does Caddy need updating on each deploy?

Usually no. Once you run tinker caddy myapp 45320 the first time, the routing rule persists — Caddy keeps routing that subdomain to that port indefinitely. Your deploy script just needs to restart the process listening on that port.

The only time you'd run tinker caddy again is if you:

  • Change the port your app runs on
  • Want to add or rename a subdomain
  • The route was accidentally removed

One-time setup vs. recurring deploy

This distinction matters. There are two categories of tasks:

One-time setup
Things you do once, manually. Clone the repo, reserve a port with tinker port, register a subdomain with tinker caddy, set up cron.
Recurring deploy
Things cron does for you automatically. Pull new code, install deps, kill old server, start new server. Caddy doesn't need to change.

Part 3 — Full setup walkthrough

Here is every step, in order, from zero to a live auto-deploying project. Steps marked one-time are done manually once. Steps marked automated are handled by your script and cron forever after.

Step 1 — Reserve a port (one-time)

tinker port
→ 45320

Write this number down. You'll use it in every command and config below.

Step 2 — Clone your repository (one-time)

# Make sure you've set up your GitHub SSH key first
# (see the SSH keys guide if you haven't)
cd ~
git clone git@github.com:yourusername/yourrepo.git
cd yourrepo

Step 3 — Write your deploy script (one-time)

Create ~/deploy.sh with the full script from Part 1 of this series. Make sure to update the APP_DIR and PORT variables to match your actual repo folder name and port number.

# Create the file
nano ~/deploy.sh

# Make it executable
chmod +x ~/deploy.sh

# Run it manually once to verify everything works
bash ~/deploy.sh

# Check the log
cat ~/deploy.log

Do not move on until this works manually. Fix any errors in the script first.

Step 4 — Start the server for the first time (one-time)

Your deploy script will start the server, but you need Caddy running before you register the subdomain. Run the deploy script manually — this starts the server on your port:

bash ~/deploy.sh

Verify the server is actually running on the correct port:

# Check that something is listening on your port
curl -s http://localhost:45320

# Or check which processes are on which ports
ss -tlnp | grep 45320
curl -s http://localhost:45320
curl makes an HTTP request from the command line. -s means silent mode (no progress bar). localhost refers to the current machine — port 45320 on itself. If your server is running, you'll see your HTML. If not, you'll get "Connection refused."
ss -tlnp
ss is "socket statistics" — shows network connections. -t = TCP, -l = listening sockets only, -n = show port numbers (not service names), -p = show which process owns each socket. Piping to grep 45320 filters to just your port.

Step 5 — Register the subdomain with Caddy (one-time)

tinker caddy myapp 45320

Wait about 30 seconds for Caddy to provision the TLS certificate, then visit https://myapp.tharavad.xyz in your browser. You should see your site over HTTPS.

If the site doesn't load immediately

TLS certificate provisioning can take up to 60 seconds on the first request. Caddy is contacting Let's Encrypt, verifying domain ownership via a DNS challenge, and downloading the certificate. Refresh after a minute. If it still doesn't work, check that your server is actually running (curl http://localhost:45320) and that your caddy route is registered (tinker caddy list).

Step 6 — Set up cron (one-time)

crontab -e

Add (replacing jake with your actual username from whoami):

crontabPATH=/usr/local/bin:/usr/bin:/bin:/home/jake/.local/bin
HOME=/home/jake

# Check for new code and redeploy every 5 minutes
*/5 * * * *  /home/jake/deploy.sh >> /home/jake/cron.log 2>&1

Save and exit. Confirm it's registered:

crontab -l

Step 7 — Test the full cycle (verification)

Push a visible change to your GitHub repo from your local machine — change a heading, add a line, anything obvious. Then watch tharavad pick it up:

# Watch the deploy log in real time
tail -f ~/cron.log

Within 5 minutes you should see your deploy script output appear in the log. Reload your subdomain in the browser. Your change should be live.

Part 4 — Handling Node.js apps

The examples so far use python3 -m http.server for simplicity, but the same pattern works for any server. Here's the deploy script adjusted for a Node.js Express app:

~/deploy.sh (Node.js version)#!/bin/bash
set -e
set -o pipefail

APP_DIR="$HOME/yourrepo"
PORT=45320
PID_FILE="/tmp/myapp.pid"
LOG_FILE="$HOME/deploy.log"
APP_LOG="$HOME/app.log"

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"; }

cd "$APP_DIR"

BEFORE=$(git rev-parse HEAD)
git fetch origin
git reset --hard origin/main
AFTER=$(git rev-parse HEAD)

if [ "$BEFORE" = "$AFTER" ]; then
    log "No changes ($(git rev-parse --short HEAD))"
    exit 0
fi

log "New code: $(git rev-parse --short HEAD). Deploying..."

# Install dependencies (only installs new/changed ones)
log "Running npm install..."
npm install --production --silent

# Stop old server
if [ -f "$PID_FILE" ]; then
    OLD_PID=$(cat "$PID_FILE")
    if kill -0 "$OLD_PID" 2>/dev/null; then
        kill "$OLD_PID" && sleep 2
        log "Stopped PID $OLD_PID"
    fi
    rm -f "$PID_FILE"
fi

# Start Node with PORT env var
PORT=$PORT node server.js >>"$APP_LOG" 2>&1 &
echo $! > "$PID_FILE"
log "Node started on port $PORT (PID $(cat $PID_FILE))"
Make sure your Node app reads PORT from the environment

In your server.js, use const port = process.env.PORT || 3000 so the deploy script can pass the port in via the PORT=45320 node server.js pattern. Never hardcode port 3000 if you want cron to control it.

Part 5 — Log rotation: keeping logs from growing forever

Your cron job runs every 5 minutes — that's 288 times a day, over 100,000 times a year. Even a short log entry per run adds up to megabytes of logs. You need to rotate them.

The simplest approach: truncate the log when it gets too big, as part of your deploy script.

# At the top of deploy.sh, after the variable definitions:
# If the log file exceeds 1MB, clear it
if [ -f "$LOG_FILE" ] && [ $(wc -c < "$LOG_FILE") -gt 1048576 ]; then
    echo "[log rotated at $(date)]" > "$LOG_FILE"
fi
wc -c < "$LOG_FILE"
wc -c counts bytes in a file. The < redirects the file content as stdin (so wc doesn't print the filename). The result is the file size in bytes.
-gt 1048576
-gt means "greater than" in bash integer comparison. 1048576 bytes = 1 megabyte (1024 × 1024).
echo "..." > "$LOG_FILE"
Single > overwrites the file (unlike >> which appends). This clears the log and writes a single "log rotated" marker line.

Part 6 — Troubleshooting

SymptomWhat to check
Site returns "502 Bad Gateway" Caddy is running but can't reach your app. Your server on port 45320 is down. Run bash ~/deploy.sh manually to restart it, then check curl http://localhost:45320.
Site not loading at all (no response) Check tinker caddy list — your subdomain may not be registered. Re-run tinker caddy myapp 45320.
Cron log is empty after 10 minutes The job may not be registered. Run crontab -l to verify. Also check for typos in the crontab — especially using ~ instead of absolute paths.
Cron log shows "command not found" Missing PATH in crontab. Make sure your crontab starts with the PATH line setting /usr/local/bin:/usr/bin:/bin.
git pull fails with "Permission denied" Your deploy key isn't set up correctly. Test with ssh -T git@github.com from your tharavad shell. If that fails, redo the GitHub SSH key setup.
Old code still showing after push Check that your deploy script does git reset --hard origin/main and not just git pull. Also verify there are no local uncommitted changes blocking the reset: git status.
Server starts but port is already in use The old process wasn't killed. Check cat /tmp/myapp.pid and then kill <pid> manually. Then run the deploy script again.
App crashes silently on start Check cat ~/app.log — this is where stdout and stderr from your running process go. Look for Python tracebacks, Node errors, missing modules, etc.
